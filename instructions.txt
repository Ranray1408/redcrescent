================================================================================
TIPTOPPAY WIDGET — ПАРАМЕТР `metadata`
Інструкція для інтеграції передачі даних з донатної сторінки до Salesforce
================================================================================

1. ЗАГАЛЬНА ІДЕЯ
--------------------------------------------------------------------------------
Параметр `metadata` у платіжному віджеті TipTopPay передає інформацію про
кампанію з лендінгової сторінки до Salesforce.

Усі ключі metadata — необов'язкові рядки; відсутні або невідомі ключі
тихо ігноруються, а неправильний payload ніколи не блокує донацію.


2. ОЧІКУВАНА СТРУКТУРА
--------------------------------------------------------------------------------
{
  publicId: "sc_48654131534684test",
  description: "Donation",
  amount: 5000,
  currency: "KZT",
  // … звичайні платіжні поля …

  metadata: {
    campaign_id:    "main",            // хардкод на кожній лендінг-сторінці
    source_code_id: "a0xAB0000004XYZ", // URL параметр
    agent_id:       "AG2317B86",       // тільки Face2Face
    venue_id:       "VN312B32"         // тільки Face2Face
  }
}


3. ОПИС ПОЛІВ
--------------------------------------------------------------------------------

3.1. campaign_id
  • Визначення: вказує на конкретну лендінг-сторінку (наприклад, main-page,
    test-campaign-2026).
  • Примітки: кампанія джерела донації — на якій сторінці був донор.
    - Salesforce шукає збіг по Campaign.core_External_Id__c.
    - Якщо знайдено — кампанія використовується повторно;
      якщо ні — створюється нова Campaign з типом запису "Donation Source".

3.2. source_code_id
  • Визначення: 18-символьний Salesforce Id існуючого запису Outreach Source Code.
  • Примітки: маркетингова кампанія — що привело донора на сторінку.
    - Salesforce шукає Outreach Source Code за Id.
    - Його батьківська Campaign стає marketing Campaign на Gift Transaction,
      Gift Commitment і Gift Commitment Schedule.
    - Сам Outreach Source Code також прив'язується.
    - Якщо Id не знайдено — маркетингові поля залишаються порожніми (без помилки).
    - Зазвичай значення береться з URL параметра ?source_code_id=.

3.3. agent_id
  • Визначення: зовнішній ідентифікатор Face2Face агента, який залучив донора
    на точці збору. Донор обирає агента з пікліста; значення, яке надсилається —
    це поле core_Face2Face_AgentId__pc вибраного агента, отримане з API
    активних агентів (Крок 1).
  • Примітки:
    - Salesforce знаходить агента за цим id (фільтруючи активних F2F агентів).
    - Агент прив'язується до Gift Transaction і Gift Commitment.
    - Для донорів-новачків це ж значення також проставляється як Source Agent
      на Person Account донора і зберігається при наступних донаціях.
    - Якщо id не відповідає жодному активному агенту — донація все одно
      проходить успішно, поля агента просто залишаються порожніми (без помилки).

3.4. venue_id
  • Визначення: зовнішній ідентифікатор Face2Face точки збору, де була
    здійснена донація. Донор обирає точку збору з пікліста; значення —
    це поле core_Face2Face_Venue_Id__c вибраної точки збору, отримане
    з API активних точок збору (Крок 1).
  • Примітки:
    - Salesforce знаходить точку збору за цим id (фільтруючи активні F2F venue).
    - Точка збору прив'язується до Gift Transaction і Gift Commitment.
    - Для донорів-новачків це ж значення проставляється як Source Venue
      на Person Account донора і зберігається при наступних донаціях.
    - Якщо id не знайдено — донація все одно проходить успішно, поля venue
      просто залишаються порожніми (без помилки).


4. ЯК ВИКОРИСТОВУВАТИ ЦЕ API
--------------------------------------------------------------------------------
Є дві форми інтеграції залежно від сторінки:

  • Face2Face лендінг — 2 запити.
    Крок 1: отримати активних агентів і точки збору з Salesforce
            (для наповнення піклістів на формі).
    Крок 2: зібрати вибір донора (включаючи обраного агента і точку збору)
            і відправити платіж у TipTopPay з цими значеннями в metadata.

  • Інші сторінки (кампанійна сторінка, відстежуване посилання, загальна) —
    1 запит. Тільки Крок 2. Надіслати платіж у TipTopPay безпосередньо
    з тими metadata, які є (наприклад, campaign_id, source_code_id).
    Крок 1 не потрібен, оскільки F2F пошуки не задіяні.

В обох випадках Salesforce читає metadata з вебхука TipTopPay і прив'язує
донацію до відповідних записів.


5. ДАНІ ДЛЯ FACE2FACE (ВІД SALESFORCE)
--------------------------------------------------------------------------------
Для Face2Face лендінгу Salesforce надасть три значення для
External Client App, який ми налаштуємо:

  • Instance URL — URL My Domain Salesforce org, наприклад:
    https://efficiency-app-7050.scratch.my.salesforce.com

  • Consumer Key — з External Client App.

  • Consumer Secret — з External Client App. Ставитися як до пароля;
    ніколи не комітити в код і не показувати в браузерному коді.


6. КРОК 1 — ОТРИМАТИ АКТИВНИХ АГЕНТІВ І ТОЧКИ ЗБОРУ З SALESFORCE
--------------------------------------------------------------------------------
Потрібен тільки для Face2Face сторінки. Виконується на бекенді (серверна
частина). Токени ніколи не потрапляють у браузер.

6.1. Отримати OAuth access token (Client Credentials flow):

  POST {instance_url}/services/oauth2/token
  Content-Type: application/x-www-form-urlencoded

  grant_type=client_credentials&client_id={CONSUMER_KEY}&client_secret={CONSUMER_SECRET}

  Відповідь містить access_token (дійсний ~2 години) та instance_url.
  Використовуйте обидва для запитів нижче. Перевикликайте цей endpoint
  коли токен прострочиться або при кожному завантаженні сторінки.

6.2. Запит активних агентів:

  GET {instance_url}/services/data/v61.0/query/?q=SELECT+Id,+Name,+core_Face2Face_AgentId__pc+FROM+Account+WHERE+IsPersonAccount=true+AND+core_Face2Face_Active_Agent__pc=true+ORDER+BY+Name

  Authorization: Bearer {access_token}

  Відповідь — JSON об'єкт з масивом records. Відображайте Name у піклісті
  агентів; запам'ятайте core_Face2Face_AgentId__pc кожного рядка —
  це значення, яке треба відправляти як agent_id у Кроці 2.

6.3. Запит активних точок збору:

  GET {instance_url}/services/data/v61.0/query/?q=SELECT+Id,+core_Face2Face_Venue_Id__c,+core_Face2Face_Venue_Name__c+FROM+Account+WHERE+IsPersonAccount=false+AND+core_Face2Face_Venue_Active__c=true+ORDER+BY+core_Face2Face_Venue_Name__c

  Authorization: Bearer {access_token}

  Відображайте core_Face2Face_Venue_Name__c у піклісті точок збору;
  запам'ятайте core_Face2Face_Venue_Id__c — це значення для venue_id у Кроці 2.

  Кешуйте відповіді на ~5-15 хвилин на сервері; списки змінюються рідко.


7. КРОК 2 — ВІДПРАВИТИ ДОНАЦІЮ В TIPTOPPAY З METADATA
--------------------------------------------------------------------------------
Коли донор надсилає форму, візьміть його вибір і помістіть значення
в об'єкт metadata, переданий у віджет TipTopPay. Включайте тільки ті ключі,
для яких у вас є значення — порожні/відсутні ключі допускаються,
але не передавайте порожні рядки.

Salesforce читає metadata назад з вебхука TipTopPay і прив'язує донацію
до відповідних записів агента, точки збору, кампанії та outreach source code.


8. СЦЕНАРІЇ ВИКЛИКІВ
--------------------------------------------------------------------------------
Кожен сценарій показує блок metadata для передачі у віджет TipTopPay.
Інші платіжні поля (publicId, amount, currency тощо) не змінюються.

8.1. Сценарій 1 — Face2Face лендінг (2 запити)

  Сторінка спочатку викликає Salesforce (Крок 1) для наповнення піклістів
  агентів і точок збору, потім надсилає донацію (Крок 2) з вибором донора
  у metadata. Той самий payload також може містити campaign_id і
  source_code_id, якщо F2F сторінка сама є кампанійною та/або на неї
  прийшли через відстежуване посилання.

  Metadata для Кроку 2:
  {
    campaign_id: "f2f-almaty",
    source_code_id: "a0xAB0000004XYZ",
    agent_id: "AG2317B86",
    venue_id: "VN312B32"
  }

  Donation Source Campaign, marketing campaign / outreach source code,
  агент і точка збору — всі прив'язуються до Gift Transaction і
  Gift Commitment. Для донорів-новачків Source Agent і Source Venue
  також проставляються на Person Account і зберігаються при
  наступних донаціях. Приберіть campaign_id та/або source_code_id,
  якщо вони не застосовуються.

  Бізнес-логіка:
  - При початковому завантаженні F2F сторінка викликає Salesforce (Крок 1),
    щоб отримати списки активних F2F агентів і точок збору.
  - Для агентів потрібен тільки ідентифікатор (core_Face2Face_AgentId__pc).
  - Для точок збору зберігається як id (core_Face2Face_Venue_Id__c),
    так і назва (core_Face2Face_Venue_Name__c).
  - Донор вводить id агента у текстове поле. Перед донацією сторінка
    перевіряє введене значення по кешованому списку id агентів з Кроку 1.
    Якщо id збігається — поле приймається. Якщо ні — показується помилка
    "Agent with id X not found" і кнопка відправки блокується.
  - Точки збору — звичайний select, наповнений зі списку Кроку 1
    (показується назва, значення — id). Вибір з закритого списку
    гарантує валідність.
  - При submit: валідований agent_id + вибраний venue_id + campaign_id
    + source_code_id → metadata → TipTopPay.

  Приклад: https://donate.redcrescent.kg/f2f

8.2. Сценарій 2 — Стандартна донатна сторінка (1 запит)

  Без F2F пошуків, тому Крок 1 пропускається. Сторінка надсилає донацію
  безпосередньо (Крок 2) з тими metadata, які застосовуються.
  Охоплює: сторінку з кампанією, відстежуване посилання на кампанійну
  сторінку, або загальну донатну сторінку без атрибуції.

  Metadata для Кроку 2:
  {
    campaign_id: "main",              // назва лендінгу
    source_code_id: "a0xAB0000004XYZ" // id Outreach Source Code з URL
  }

  Donation Source Campaign встановлюється з campaign_id. Якщо включено
  source_code_id — донація також зв'язується з маркетинговою кампанією
  та outreach source code. Пропустіть metadata повністю (або передайте {}),
  якщо це загальна донатна сторінка без атрибуції.

  Бізнес-логіка:
  - Жодного Salesforce запиту при завантаженні — немає піклістів агентів
    або точок збору. Сторінка просто збирає платіжні дані донора і
    надсилає один TipTopPay платіж (тільки Крок 2).
  - Будь-яка атрибуція (campaign_id, source_code_id) надається самою
    сторінкою — хардкод для лендінгу, або зчитана з URL параметрів
    відстежуваного посилання — і передається в metadata без змін.
    Донор ніколи не вводить ці значення.

  Приклад: https://donate.redcrescent.kg/


9. ЩО РОБИТИ, ЯКЩО ЗНАЧЕННЯ НЕВІДОМЕ?
--------------------------------------------------------------------------------
Пропустіть ключ повністю. Не передавайте null, порожній рядок або
заповнювач. Інтеграція стійка до помилок — відсутній ключ просто
залишає відповідне поле Salesforce порожнім, без помилки,
донація все одно проходить.


10. РЕКУРЕНТНІ ПЛАТЕЖІ (ПІДПИСКА ПІД УПРАВЛІННЯМ SALESFORCE)
--------------------------------------------------------------------------------
Для рекурентних платежів Salesforce керує графіком підписки, а не TipTopPay.
TipTopPay обробляє тільки початковий платіж як одноразовий токенізований
платіж; кожен наступний рекурентний платіж ініціюється Salesforce у
щоденному batch-запуску з використанням збереженого карткового токена.

10.1. Зміни у віджеті для рекурентних платежів:

  • Перемістіть об'єкт recurrent з основного тіла запиту в параметр
    metadata, ключем recurrent. Це дискримінатор, який Salesforce
    використовує для розпізнавання підписки, керованої Salesforce,
    у вхідному Pay webhook.

  • Додайте tokenize: true до основного тіла запиту. Це вказує
    TipTopPay повернути поле Token у Pay-нотифікації, щоб Salesforce
    міг списувати кошти з картки за графіком.

  • НЕ залишайте recurrent в основному тілі запиту. Якщо він
    залишиться в тілі, TipTopPay створить власну підписку на своєму
    боці — чого ми не хочемо (див. нижче).

10.2. Чому ця зміна потрібна:

  Вбудована рекурентна підписка TipTopPay автоматично скасовує підписку
  донора після 3 невдалих спроб поспіль. Ми не хочемо такої поведінки —
  донори не повинні втрачати підписку після трьох тимчасових збоїв
  (недостатньо коштів, тайм-аут емітента тощо). Salesforce-керований
  шлях передає рішення про повторну спробу в Salesforce, яка застосовує
  власну політику повторів і припиняє підписку тільки коли причина збою
  є остаточною (втрачена / вкрадена / прострочена картка, фрод-блок,
  обмежена картка тощо).

10.3. Очікувана структура (рекурентний платіж, Крок 2):

  {
    publicId: "sc_48654131534684test",
    description: "Donation",
    amount: 5000,
    currency: "KZT",
    tokenize: true,                      // обов'язково для SF-керованої підписки
    // … звичайні платіжні поля …
    metadata: {
      campaign_id: "main",
      source_code_id: "a0xAB0000004XYZ",
      recurrent: {                        // ПЕРЕМІЩЕНО сюди з основного тіла
        interval: "Month",
        period: 1,
        amount: 5000,
        startDate: "2026-06-15T00:00:00.0000000Z"
      }
    }
  }

10.4. Обмеження (поточний обсяг):
  • recurrent.interval — завжди "Month".
  • recurrent.period — завжди 1.
  • recurrent.startDate (ISO 8601) — якір, що визначає день місяця
    для наступних списань. Використовуйте сьогоднішню дату.
  • recurrent.amount — передавайте ту саму суму, яку вибрав донор
    у формі.
