================================================================================
TIPTOPPAY WIDGET — ПАРАМЕТР `metadata`
Инструкция по интеграции передачи данных со страницы пожертвований в Salesforce
================================================================================

1. ОБЩАЯ ИДЕЯ
--------------------------------------------------------------------------------
Параметр `metadata` в платёжном виджете TipTopPay передаёт информацию о
кампании с лендинговой страницы в Salesforce.

Все ключи metadata — необязательные строки; отсутствующие или неизвестные
ключи молча игнорируются, а некорректный payload никогда не блокирует пожертвование.


2. ОЖИДАЕМАЯ СТРУКТУРА
--------------------------------------------------------------------------------
{
  publicId: "sc_48654131534684test",
  description: "Donation",
  amount: 5000,
  currency: "KZT",
  // … обычные платёжные поля …

  metadata: {
    campaign_id:    "main",            // хардкод на каждой лендинг-странице
    source_code_id: "a0xAB0000004XYZ", // URL параметр
    agent_id:       "AG2317B86",       // только Face2Face
    venue_id:       "VN312B32"         // только Face2Face
  }
}


3. ОПИСАНИЕ ПОЛЕЙ
--------------------------------------------------------------------------------

3.1. campaign_id
  • Определение: указывает на конкретную лендинг-страницу (например,
    main-page, test-campaign-2026).
  • Примечания: кампания-источник пожертвования — на какой странице
    находился донор.
    - Salesforce ищет совпадение по Campaign.core_External_Id__c.
    - Если найдено — кампания используется повторно;
      если нет — создаётся новая Campaign с типом записи "Donation Source".

3.2. source_code_id
  • Определение: 18-символьный Salesforce Id существующей записи
    Outreach Source Code.
  • Примечания: маркетинговая кампания — что привело донора на страницу.
    - Salesforce ищет Outreach Source Code по Id.
    - Его родительская Campaign становится marketing Campaign на
      Gift Transaction, Gift Commitment и Gift Commitment Schedule.
    - Сам Outreach Source Code также привязывается.
    - Если Id не найден — маркетинговые поля остаются пустыми (без ошибки).
    - Обычно значение берётся из URL параметра ?source_code_id=.

3.3. agent_id
  • Определение: внешний идентификатор Face2Face агента, который привлёк
    донора в точке сбора. Донор выбирает агента из пиклиста; отправляемое
    значение — это поле core_Face2Face_AgentId__c выбранного агента,
    полученное из API активных агентов (Шаг 1).
  • Примечания:
    - Salesforce находит агента по этому id (фильтруя активных F2F агентов).
    - Агент привязывается к Gift Transaction и Gift Commitment.
    - Для новых доноров это же значение также проставляется как Source Agent
      на Person Account донора и сохраняется при последующих пожертвованиях.
    - Если id не соответствует ни одному активному агенту — пожертвование
      всё равно проходит успешно, поля агента остаются пустыми (без ошибки).

3.4. venue_id
  • Определение: внешний идентификатор Face2Face точки сбора, где было
    совершено пожертвование. Донор выбирает точку сбора из пиклиста;
    значение — это поле core_Face2Face_Venue_Id__c выбранной точки сбора,
    полученное из API активных точек сбора (Шаг 1).
  • Примечания:
    - Salesforce находит точку сбора по этому id (фильтруя активные F2F venue).
    - Точка сбора привязывается к Gift Transaction и Gift Commitment.
    - Для новых доноров это же значение проставляется как Source Venue
      на Person Account донора и сохраняется при последующих пожертвованиях.
    - Если id не найден — пожертвование всё равно проходит успешно,
      поля venue остаются пустыми (без ошибки).


4. КАК ИСПОЛЬЗОВАТЬ ЭТО API
--------------------------------------------------------------------------------
Есть две формы интеграции в зависимости от страницы:

  • Face2Face лендинг — 2 запроса.
    Шаг 1: получить активных агентов и точки сбора из Salesforce
            (для наполнения пиклистов на форме).
    Шаг 2: собрать выбор донора (включая выбранного агента и точку сбора)
            и отправить платёж в TipTopPay с этими значениями в metadata.

  • Другие страницы (кампанийная страница, отслеживаемая ссылка,
    общая страница) — 1 запрос. Только Шаг 2. Отправить платёж в TipTopPay
    напрямую с теми metadata, которые есть (например, campaign_id,
    source_code_id). Шаг 1 не нужен, так как F2F поиски не задействованы.

В обоих случаях Salesforce читает metadata из вебхука TipTopPay и привязывает
пожертвование к соответствующим записям.


5. ДАННЫЕ ДЛЯ FACE2FACE (ОТ SALESFORCE)
--------------------------------------------------------------------------------
Для Face2Face лендинга Salesforce предоставит три значения для
External Client App, который мы настроим:

  • Instance URL — URL My Domain Salesforce org, например:
    https://efficiency-app-7050.scratch.my.salesforce.com

  • Consumer Key — из External Client App.

  • Consumer Secret — из External Client App. Относиться как к паролю;
    никогда не коммитить в код и не показывать в браузерном коде.


6. ШАГ 1 — ПОЛУЧИТЬ АКТИВНЫХ АГЕНТОВ И ТОЧКИ СБОРА ИЗ SALESFORCE
--------------------------------------------------------------------------------
Требуется только для Face2Face страницы. Выполняется на бэкенде (серверная
часть). Токены никогда не попадают в браузер.

6.1. Получить OAuth access token (Client Credentials flow):

  POST {instance_url}/services/oauth2/token
  Content-Type: application/x-www-form-urlencoded

  grant_type=client_credentials&client_id={CONSUMER_KEY}&client_secret={CONSUMER_SECRET}

  Ответ содержит access_token (действителен ~2 часа) и instance_url.
  Используйте оба для запросов ниже. Перевызывайте этот endpoint
  когда токен истечёт или при каждой загрузке страницы.

6.2. Запрос активных агентов:

  GET {instance_url}/services/data/v61.0/query/?q=SELECT+Id,+Name,+core_Face2Face_AgentId__pc+FROM+Account+WHERE+IsPersonAccount=true+AND+core_Face2Face_Active_Agent__pc=true+ORDER+BY+Name

  Authorization: Bearer {access_token}

  Ответ — JSON объект с массивом records. Отображайте Name в пиклисте
  агентов; запомните core_Face2Face_AgentId__pc каждой строки —
  это значение нужно отправлять как agent_id на Шаге 2.

6.3. Запрос активных точек сбора:

  GET {instance_url}/services/data/v61.0/query/?q=SELECT+Id,+core_Face2Face_Venue_Id__c,+core_Face2Face_Venue_Name__c+FROM+Account+WHERE+IsPersonAccount=false+AND+core_Face2Face_Venue_Active__c=true+ORDER+BY+core_Face2Face_Venue_Name__c

  Authorization: Bearer {access_token}

  Отображайте core_Face2Face_Venue_Name__c в пиклисте точек сбора;
  запомните core_Face2Face_Venue_Id__c — это значение для venue_id на Шаге 2.

  Кешируйте ответы на ~5-15 минут на сервере; списки меняются редко.


7. ШАГ 2 — ОТПРАВИТЬ ПОЖЕРТВОВАНИЕ В TIPTOPPAY С METADATA
--------------------------------------------------------------------------------
Когда донор отправляет форму, возьмите его выбор и поместите значения
в объект metadata, переданный в виджет TipTopPay. Включайте только те ключи,
для которых у вас есть значение — пустые/отсутствующие ключи допускаются,
но не передавайте пустые строки.

Salesforce читает metadata обратно из вебхука TipTopPay и привязывает
пожертвование к соответствующим записям агента, точки сбора, кампании
и outreach source code.


8. СЦЕНАРИИ ВЫЗОВОВ
--------------------------------------------------------------------------------
Каждый сценарий показывает блок metadata для передачи в виджет TipTopPay.
Остальные платёжные поля (publicId, amount, currency и т.д.) не меняются.

8.1. Сценарий 1 — Face2Face лендинг (2 запроса)

  Страница сначала вызывает Salesforce (Шаг 1) для наполнения пиклистов
  агентов и точек сбора, затем отправляет пожертвование (Шаг 2) с выбором
  донора в metadata. Тот же payload также может содержать campaign_id и
  source_code_id, если F2F страница сама является кампанийной и/или на неё
  пришли через отслеживаемую ссылку.

  Metadata для Шага 2:
  {
    campaign_id: "f2f-almaty",
    source_code_id: "a0xAB0000004XYZ",
    agent_id: "AG2317B86",
    venue_id: "VN312B32"
  }

  Donation Source Campaign, marketing campaign / outreach source code,
  агент и точка сбора — все привязываются к Gift Transaction и
  Gift Commitment. Для новых доноров Source Agent и Source Venue
  также проставляются на Person Account и сохраняются при
  последующих пожертвованиях. Уберите campaign_id и/или source_code_id,
  если они не применяются.

  Бизнес-логика:
  - При начальной загрузке F2F страница вызывает Salesforce (Шаг 1),
    чтобы получить списки активных F2F агентов и точек сбора.
  - Для агентов нужен только идентификатор (core_Face2Face_AgentId__pc).
  - Для точек сбора сохраняется как id (core_Face2Face_Venue_Id__c),
    так и название (core_Face2Face_Venue_Name__c).
  - Донор вводит id агента в текстовое поле. Перед пожертвованием
    страница проверяет введённое значение по кешированному списку
    id агентов с Шага 1. Если id совпадает — поле принимается.
    Если нет — показывается ошибка "Agent with id X not found"
    и кнопка отправки блокируется.
  - Точки сбора — обычный select, наполненный из списка Шага 1
    (показывается название, значение — id). Выбор из закрытого
    списка гарантирует валидность.
  - При submit: валидированный agent_id + выбранный venue_id +
    campaign_id + source_code_id → metadata → TipTopPay.

  Пример: https://donate.redcrescent.kg/f2f

8.2. Сценарий 2 — Стандартная страница пожертвования (1 запрос)

  Без F2F поисков, поэтому Шаг 1 пропускается. Страница отправляет
  пожертвование напрямую (Шаг 2) с теми metadata, которые применяются.
  Охватывает: страницу с кампанией, отслеживаемую ссылку на кампанийную
  страницу, или общую страницу пожертвования без атрибуции.

  Metadata для Шага 2:
  {
    campaign_id: "main",              // название лендинга
    source_code_id: "a0xAB0000004XYZ" // id Outreach Source Code из URL
  }

  Donation Source Campaign устанавливается из campaign_id. Если включён
  source_code_id — пожертвование также связывается с маркетинговой
  кампанией и outreach source code. Опустите metadata полностью
  (или передайте {}), если это общая страница пожертвования без атрибуции.

  Бизнес-логика:
  - Никакого Salesforce запроса при загрузке — нет пиклистов агентов
    или точек сбора. Страница просто собирает платёжные данные донора
    и отправляет один TipTopPay платёж (только Шаг 2).
  - Любая атрибуция (campaign_id, source_code_id) предоставляется самой
    страницей — хардкод для лендинга, или считана из URL параметров
    отслеживаемой ссылки — и передаётся в metadata без изменений.
    Донор никогда не вводит эти значения.

  Пример: https://donate.redcrescent.kg/


9. ЧТО ДЕЛАТЬ, ЕСЛИ ЗНАЧЕНИЕ НЕИЗВЕСТНО?
--------------------------------------------------------------------------------
Опустите ключ полностью. Не передавайте null, пустую строку или
заполнитель. Интеграция устойчива к ошибкам — отсутствующий ключ
просто оставляет соответствующее поле Salesforce пустым, без ошибки,
пожертвование всё равно проходит.


10. РЕКУРРЕНТНЫЕ ПЛАТЕЖИ (ПОДПИСКА ПОД УПРАВЛЕНИЕМ SALESFORCE)
--------------------------------------------------------------------------------
Для рекуррентных платежей Salesforce управляет графиком подписки, а не
TipTopPay. TipTopPay обрабатывает только начальный платёж как одноразовый
токенизированный платёж; каждый последующий рекуррентный платёж
инициируется Salesforce в ежедневном batch-запуске с использованием
сохранённого карточного токена.

10.1. Изменения в виджете для рекуррентных платежей:

  • Переместите объект recurrent из основного тела запроса в параметр
    metadata, ключом recurrent. Это дискриминатор, который Salesforce
    использует для распознавания подписки, управляемой Salesforce,
    во входящем Pay webhook.

  • Добавьте tokenize: true в основное тело запроса. Это указывает
    TipTopPay вернуть поле Token в Pay-уведомлении, чтобы Salesforce
    мог списывать средства с карты по графику.

  • НЕ оставляйте recurrent в основном теле запроса. Если он
    останется в теле, TipTopPay создаст собственную подписку на своей
    стороне — чего мы не хотим (см. ниже).

10.2. Почему это изменение нужно:

  Встроенная рекуррентная подписка TipTopPay автоматически отменяет
  подписку донора после 3 неудачных попыток подряд. Мы не хотим такого
  поведения — доноры не должны терять подписку после трёх временных
  сбоев (недостаточно средств, тайм-аут эмитента и т.д.).
  Salesforce-управляемый путь передаёт решение о повторной попытке в
  Salesforce, которая применяет собственную политику повторов и
  прекращает подписку только когда причина сбоя является окончательной
  (утраченная / украденная / просроченная карта, фрод-блок,
  ограниченная карта и т.д.).

10.3. Ожидаемая структура (рекуррентный платёж, Шаг 2):

  {
    publicId: "sc_48654131534684test",
    description: "Donation",
    amount: 5000,
    currency: "KZT",
    tokenize: true,                      // обязательно для SF-управляемой подписки
    // … обычные платёжные поля …
    metadata: {
      campaign_id: "main",
      source_code_id: "a0xAB0000004XYZ",
      recurrent: {                        // ПЕРЕНЕСЕНО сюда из основного тела
        interval: "Month",
        period: 1,
        amount: 5000,
        startDate: "2026-06-15T00:00:00.0000000Z"
      }
    }
  }

10.4. Ограничения (текущий объём):
  • recurrent.interval — всегда "Month".
  • recurrent.period — всегда 1.
  • recurrent.startDate (ISO 8601) — якорь, определяющий день месяца
    для последующих списаний. Используйте сегодняшнюю дату.
  • recurrent.amount — передавайте ту же сумму, которую выбрал донор
    в форме.
